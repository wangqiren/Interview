##### 自动装箱的和拆箱的底层原理

自动装箱的底层是通过调用包装类的静态工厂方式valueof()实现的

自动拆箱的底层是通过调用包装类的实例方法如intValue()实现的

##### Integer缓存范围是多少

```java
Integer a = 100;
Integer b = 100;
System.out.println(a == b); // 输出 true

Integer x = 200;
Integer y = 200;
System.out.println(x == y); // 输出 false
```

为什么同样是 `==` 比较，结果却不同呢？原因就在 `Integer.valueOf()` 方法的实现里。

**Integer.java的源码片段**

```java
public static Integer valueOf(int i) {
    if (i >= IntegerCache.low && i <= IntegerCache.high)
        return IntegerCache.cache[i + (-IntegerCache.low)];
    return new Integer(i);
}
```

当你进行自动装箱时，它会先判断int值是否在一个预设的范围内。如果在，就直接从缓存数组IntegerCache.cache中返回一个预先创建好的，已经存在的对象，如果不在，才new一个新的对象

**IntegerCache的实现**

```java
private static class IntegerCache {
    static final int low = -128;
    static final int high;
    static final Integer cache[];

    static {
        // high value may be configured by property
        int h = 127;
        // 省略... 读取JVM参数来配置high值的逻辑
        high = h;

        cache = new Integer[(high - low) + 1];
        int j = low;
        for(int k = 0; k < cache.length; k++)
            cache[k] = new Integer(j++); // 提前创建好所有对象
    }
}
```

- 默认范围：low固定是-128，high默认是127。因此默认缓存了【-128,127】这个闭区间的所有整数对象
- 可配置上限：VM提供了启动参数 `-XX:AutoBoxCacheMax=<size>` 或 `-Djava.lang.Integer.IntegerCache.high=<value>` 来修改缓存的上限值
- 初始化时机：这个缓存是在 `Integer` 类首次被加载（第一次使用）时，通过静态代码块一次性初始化完成的。

##### 为什么这么设计

1. **“局部性原理”与常用性**：在绝大多数Java程序中，-128到127之间的小整数（如状态码、ID、年龄、索引等）是使用频率最高的。缓存这些最常用的值，可以显著提升性能。
2. **减少对象创建，节约内存**：包装类是不可变类，每次 `new` 都会在堆上分配新内存。如果在一个循环或高频调用的方法中频繁使用 `Integer`，会创建大量值相同但对象不同的实例，给垃圾回收（GC）带来巨大压力。通过复用缓存中的对象，可以极大地减少内存分配和GC开销。
3. **JLS规范要求**：Java语言规范（Java Language Specification, JLS）的**装箱转换（Boxing Conversion）** 章节中，也要求对于在 -128 到 127 之间的 `int` 字面量进行装箱时，必须保证复用相同对象。这不仅仅是JVM的实现细节，更是语言层面的约定。

#### 为什么重写equals时必须重写hashCode？

##### equals()与hashcode()的默认行为

- Object.equals()：默认实现使用==比较两个对象的内存地址，即只有当两个引用执行同一个对象时才返回true
- Object.hashCode()：默认实现返回一个由JVM根据对象内存地址计算出的整数；通常是将内存地址转换成整数，但并非绝对

默认情况下，这两个方法是一致的：两个不同的对象（不同地址）有不同的哈希码

##### 为什么需要重写equals()

当我们需要**逻辑相等**而非**引用相等**时，就必须重写equals()

例如，有一个User类，你认为只要两个User对象的userId相同，它就代表同一个用户，无论他们是不是同一个内存对象

```java
public class User {
    private String userId;
    private String name;
    // getters, setters...
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        User user = (User) o;
        return Objects.equals(userId, user.userId);
    }
}
```

此时如果只重写 `equals()` 而不重写 `hashCode()`，就会违反 `hashCode` 的通用约定。

##### hashcode()的通用约定

1. **一致性**：在java应用的一次执行过程中，对同一个对象多次调用hashcode()必须返回相同的整数。（前提是equals比较中使用的信息未被修改）
2. **相等对象必须具有相等的哈希码**：如果两个对象根据 `equals(Object)` 比较相等，那么对这两个对象调用 `hashCode()` 必须产生相同的整数结果。
3. **不相等对象不要求哈希码不同**：如果两个对象根据 `equals` 不相等，它们的哈希码可以相同（哈希碰撞），但为提升性能，最好尽量不同

其中第二条是核心：当两个对象罗技相等时，他们的哈希码必须相等

##### 如果不重写hashCode会怎样

HashSet添加元素时，首先调用对象的hashCode()计算存储位置（桶索引）

user1和user2的hashCode(不同)，他们被放入不同的桶中

##### 为什么违反约定的为题如此严重

基于哈希的集合（HashMap，HashSet，ConcurrentMap）的工作原理是：

1. 使用hashCode()定位到存储桶
2. 如果桶内有多个元素，再使用equals()再桶内线性比较找到目标

如果不重写hashCode()，逻辑相等的对象会落在不同桶中，导致：

- 重复元素：集合无法去重
- 无法查找：即使对象存在，也可能查找失败
- 内存浪费：同一逻辑对象存储多份
- 性能下降：极端情况下，所有对象落到不同桶中，哈希表退化成链表或树，失去性能

#### String、StringBuffer、StringBuilder的区别？String为什么被设计成不可变类？

| 特性           | String                                     | StringBuffer             | StringBuilder                          |
| -------------- | ------------------------------------------ | ------------------------ | -------------------------------------- |
| **可变性**     | 不可变                                     | 可变                     | 可变                                   |
| **线程安全性** | 线程安全                                   | 线程安全                 | 线程不安全                             |
| **性能**       | 拼接、修改等操作会产生大量临时对象，效率低 | 较低（有同步开销）       | 最高                                   |
| **引入版本**   | Java 1.0                                   | Java 1.0                 | Java 1.5                               |
| **适用场景**   | 字符串常量、少量操作或不需要修改的场景     | 多线程环境下的字符串操作 | 单线程环境下的字符串拼接、修改（循环） |

##### 可变性

- String类内部使用 `private final char value[]` 存储字符 （Java 9+ 改为`byte[]`并支持编码标识）且没有提供任何修改内部数组的方法。任何对String的修改（如concat，replace）都会返回一个新的String对象，原对象不变
- StringBuffer和StringBuilder继承自AbstractBuilder，他们内部也是字符数组，但提供了append，insert，delete等方法直接修改该数组，因此是可变的

##### 线程安全性

- String的不可变性决定了它可以在多个线程之间安全共享，无需同步
- StringBUffer的所有public方法都使用`synchronized`关键字修饰，保证了线程安全，但带来了性能损耗
- StringBuilder的方法没有同步，因此在单线程环境下性能更高，但在多线程并发修改时可能导致数据不一致

##### 性能对比

- 对于简单的字符串拼接，`String`使用 `+` 操作符时，编译器会优化为`StringBuilder`
- StringBuilder最快，直接操作内部数组，无同步开销
- StringBuffer次之，因为带来额外开销

##### 线程私有的

##### 线程共享的

Java虚拟机规范对于运行时数据区域的规定是相当宽松的

以堆为例：堆可以是连续空间，也可以不连续。堆的大小可以固定，也可以在运行时按需扩展。虚拟机实现者可以使用任何垃圾回收器算法管理堆，甚至完全不进行垃圾收集也是可以的

#### 程序计数器

##### 核心特性

- 线程私有/独立存储
- 较小内存空间

##### 主要功能

- 代码流程控制：分支、循环、一场
- 线程恢复：记录切换位置

##### 执行状态

- Java方式：记录字节码指令地址
- Native方法：undefined

##### 声明周期与异常

- 随线程创建而创建/销毁
- 唯一不报OutOfMemoryError区域

##### 详细介绍

程序计数器是一块较小的内存空间，可以看作是当前线程所执行的字节码的行号指示器。字节码解释器工作时通过改变这个计数器的值来选取下一条需要执行的字节码指令，分支、循环、跳转、异常处理、线程恢复等功能都需要依赖这个计数器完成

另外，为了线程切换后能恢复到正确的执行位置，每条线程都需要有一个独立的程序计数器，各线程之间计数器互不影响，独立存储，我们称这类内存区域为`线程私有`的内存

从上面的介绍中我们知道了程序计数器主要有两个作用：

- 字节码解释器通过改变程序计数器来依次读取指令，从而实现代码的流程控制，如：顺序执行、选择、循环、异常处理
- 在多线程的情况下，程序计数器用于记录当前线程执行的位置，从而当线程被切换回来的时候能够知道该线程上次运行到哪儿

程序计数器的生命周期与线程完全同步：

- 创建：随着线程的创建而创建
- 销毁：随着线程的结束而销毁

在执行Java方法（非native）时，程序计数器记录的是当前正在执行的JVM字节码指令的地址。当线程执行的是一个native方法（本地方法）时，程序计数器的值为Undedined（未定义）。这是因为native方法不执行JVM字节码，而是通过JNI调用本地平台的底层代码，JVM无需再跟踪字节码地址。

> [!CAUTION]
>
> 程序计数器是JVM规范中唯一没有规定任何`OutofMemoryError`情况的内存区域。这是因为它的内存占用极小且固定，不会出现内存溢出的情况

#### Java虚拟机栈

##### 核心特性

- 线程私有，随线程创建/销毁
- 服务对象：Java方法
- 栈针先进后出

##### 栈针结构

局部变量表、操作数栈、动态连接、出口信息

##### 异常情况

- StackOverflowError：栈深度溢出
- OutOfMemoryError：内存扩展失败

与升序计数器一样，Java虚拟机栈（后简称栈）也是线程私有的，它的生命周期和线程相同，随着线程的创建而创建，随着线程的死亡而死亡

Java虚拟机栈绝对算的上是JVM运行时数据区域的一个核心，除了一些Native方法调用是通过本地方法栈实现的，其他的所有的Java方法调用都是通过栈来实现的（也需要和其他运行时数据区域比如程序计数器配合）

方法嗲用的数据需要通过栈进行传递，每一次方法调用都会有一个对应的栈针被压入栈中，每一个方法嗲用结束后，都会有一个栈针被弹出

栈由一个个栈针组成，而每个栈针中都拥有：局部变量表、操作数栈、动态链接、方法返回地址。和数据结构上的栈类似，两者都是先进先出的数据结构，只支持出栈和入栈两种操作

`局部变量表`主要存放了编译期可知的各种数据类型（boolean、byte、char、short、int、float、long、double）、对象引用（reference类型，它不同于对象本身，可能是一个指向对象起始地址的引用指针，也可能是指向一个代表对象的句柄或其他与此对象相关的位置）。

`操作数栈`主要作为方法调用的中转站使用，用于存放方法执行过程中产生的中间计算结果。另外，计算过程中产生的临时变量也会放在操作数栈中

`动态链接`是Java虚拟机实现方法调用的关键机制之一。在Class文件中，方法调用以`符号引用`的形式存在于常量池。为了执行调用，这些符号引用必须被转换为内存中的`直接引用`。这个转换过程分为两种情况：对于静态方法、私有方法等在编译期就能确定版本的方法，这个转换在`类加载的解析阶段`就完成了，这称为`静态解析`。而对于需要根据对象实际类型才能确定具体实现的`虚方法`（这是实现多态的基础），这个转换过程则被推迟到`程序运行期间`，由`动态链接`来完成。因此，`动态链接`的核心作用是在`在运行时解析虚方法的调用点，将其链接到正确的方法版本上`

![](https://raw.githubusercontent.com/wangqiren/Image/main/20260315095743667.png)

栈空间虽然不是无限的，但一般正常调用的情况下是不会出现问题的。不过，如果函数调用陷入无限循环的话，就会导致栈中被压入太多栈帧而占用太多空间，导致栈空间过深。那么当线程请求栈的深度超过当前 Java 虚拟机栈的最大深度的时候，就抛出 `StackOverFlowError` 错误。

`Java方法有两种返回方式：`

- 正常返回：执行return语句，返回值传递给调用者
- 异常范湖：方法执行过程中抛出异常且未被捕获

不管哪种返回方式，都会导致栈帧被弹出。也就是说， **栈帧随着方法调用而创建，随着方法结束而销毁。无论方法正常完成还是异常完成都算作方法结束。**

除了 `StackOverFlowError` 错误之外，栈还可能会出现`OutOfMemoryError`错误，这是因为如果栈的内存大小可以动态扩展， 那么当虚拟机在动态扩展栈时无法申请到足够的内存空间，则抛出`OutOfMemoryError`异常。

简单总结一下程序运行中栈可能会出现两种错误：

- **`StackOverFlowError`：** 如果栈的内存大小不允许动态扩展，那么当线程请求栈的深度超过当前 Java 虚拟机栈的最大深度的时候，就抛出 `StackOverFlowError` 错误。
- **`OutOfMemoryError`：** 如果栈的内存大小可以动态扩展， 那么当虚拟机在动态扩展栈时无法申请到足够的内存空间，则抛出`OutOfMemoryError`异常。

#### 本地方法栈

##### 定义与对比

- 作用与虚拟机栈相似
- 服务对象：Native方法

##### 虚拟机实现

HotSpot与虚拟机栈合二为一

##### 栈针内容

局部变量表、操作数栈、动态链接、出口信息

##### 异常与内存

- StackOverflowError：栈深度溢出
- OutOfMemoryError：内存扩展失败

和虚拟机栈所发挥的作用非常相似，区别是：**虚拟机栈为虚拟机执行 Java 方法 （也就是字节码）服务，而本地方法栈则为虚拟机使用到的 Native 方法服务。** 在 HotSpot 虚拟机中和 Java 虚拟机栈合二为一。

本地方法被执行的时候，在本地方法栈也会创建一个栈帧，用于存放该本地方法的局部变量表、操作数栈、动态链接、出口信息。

方法执行完毕后相应的栈帧也会出栈并释放内存空间，也会出现 `StackOverFlowError` 和 `OutOfMemoryError` 两种错误。

#### 堆

##### 定义与地位

- JVM内存中最大区域
- 所有线程共享
- 虚拟机启动时创建，生命周期长

##### 核心用途

- 存放对象实例（非静态字段）
- 存放数组数据
- 对象内存统一管理

##### 分代结构

- 新生代：Eden区 + 两个Survivor区
- 老年代：Old Generation
- 目的：优化垃圾回收效率

Java虚拟机所管理的内存中最大的一块，Java堆事所有线程共享的一块内存区域，在虚拟机启动时创建。**此内存区域的唯一目的就是存放对象实例，几乎所有的对象实例以及数组都在这里分配内存**

Java世界中**几乎**所有的对象都在堆中分配，但是，随着JIT编译器的发展与逃逸分析技术逐渐成熟，栈上分配、标量替换优化技术将会导致一些微妙的变化，所有的对象都分配到堆上也渐渐变得不那么绝对了。从JDK1.7开始已经默认开启逃逸分析，如果某些方法中的对象引用没有被返回或者未被外面使用（也就是未逃逸出去），那么对象可以直接在栈上分配内存

Java堆是垃圾收集器管理的主要区域，因此也被称作GC堆。从垃圾回收的角度，由于现在收集器基本都采用分代垃圾收集算法，所有Java堆还可以细分为：新生代和老年代；再细致一点有Eden、Survivor、Old等空间。进一步划分的目的是更好地回收内存，或者更快地分配内存。

再JDK7版本及JDK7版本之前，堆内存通常被分为下面三部分：

- 新生代内存
- 老生代
- 永久代

**JDK 8 版本之后 PermGen(永久代) 已被 Metaspace(元空间) 取代，元空间使用的是本地内存**

大部分情况，对象都会首先再Eden区域分配，在一次新生代垃圾回收后，如果对象还存活，则会进入S0或者S1，丙炔对象的年龄还会加1（Eden区->Survivor区后对象的初始年龄变为1），当它的年龄增加到一定程度（默认15岁），就会被晋升到老年代中。对象晋升到老年代的年龄阈值，可以通过参数`-XX:MaxTenuringThreshold`来设置。

##### **为什么年龄只能是 0-15?**

因为记录年龄的区域在对象头中，这个区域的大小通常是4位。这4位可以表示的最大二进制数字是1111，即十进制的15.因此，对象的年龄被限制位0到15

> [!TIP]
>
> 在HotSpot虚拟机中，对象在内存中存储的布局可以分为3块区域：对象头、实例数据和对齐填充。其中，对象头包括两部分：标记字段和类型指针。
>
> 这个年龄信息就是在标记字段中存放的（标记字段还存放了对象自身的其他信息比如哈希码、锁状态信息等）

#### 方法区

##### 定义与地位

- 线程共享的内存区域
- JVM规范定义的逻辑区域
- 具体实现随虚拟机而异

##### 核心存储内容

- 类的元数据：结构、字段、方法信息
- 方法的字节码：原始指令序列
- 运行时常量池：字面量与符号引用

##### 位置演变与例外

- 静态变量：移至Java堆
- 字符换常量池：移至Java堆
- JIT代码缓存：独立Code Cache区域

方法区属于是JVM运行时数据区域的一块逻辑区域，是各个线程共享的内存区域

《Java虚拟机规范》只是规定了有方法去这么个概念和作用，方法区到底要如何实现那就是虚拟机自己要考虑的事情了。也就是说，在不同的虚拟机实现上，方法区的实现是不同的

当虚拟机加载一个类时，它会从Class文件中解析出相应的信息，并将这些元数据存入方法区。具体来说，方法区主要存储以下核心数据：

1. 类的元数据：包括类的完整结构，如类名、父类、实现的接口、访问修饰符，以及字段和方法的详细信息（名称、类型、修饰符等）
2. 方法的字节码：每个方法的原始指令序列
3. 运行时常量池：每个类独有的，由 Class 文件中的常量池转换而来，用于存放编译期生成的各种字面量和对类型、字段、方法的符号引用。

需要特别注意的是，以下几类数据虽然在逻辑上与类相关，但在 HotSpot 虚拟机中，它们并不存储在方法区内：

- **静态变量（Static Variables）**：自 JDK 7 起，静态变量已从方法区（永久代）**移至 Java 堆（Heap）中**，与该类的 `java.lang.Class` 对象一起存放。
- **字符串常量池（String Pool）**：同样自 JDK 7 起，字符串常量池也**移至 Java 堆中**。
- **即时编译器编译后的代码缓存（JIT Code Cache）**：JIT 编译器将热点方法的字节码编译成的本地机器码，存放在一个**独立的、名为“Code Cache”的内存区域**，而不是方法区本身。这样做是为了实现更高效的执行和内存管理。

##### 方法区和永久代以及圆孔是什么关系呢

方法区额永久代以及元空间的关系很像Java中接口和类的关系，类实现了接口，这里的类就可以看做是永久代和元空间，接口可以看作是方法区，也就是说永久代以及元空间是HotSpot虚拟机对虚拟机规范中方法区的两种实现方式。并且永久代是 JDK 1.8 之前的方法区实现，JDK 1.8 及以后方法区的实现变成了元空间

##### 为什么要将永久代 (PermGen) 替换为元空间 (MetaSpace) 呢?

1、整个永久代有一个 JVM 本身设置的固定大小上限，无法进行调整（也就是受到 JVM 内存的限制），而元空间使用的是本地内存，受本机可用内存的限制，虽然元空间仍旧可能溢出，但是比原来出现的几率会更小。

> 当元空间溢出时会得到如下错误：`java.lang.OutOfMemoryError: MetaSpace`

你可以使用 `-XX：MaxMetaspaceSize` 标志设置最大元空间大小，默认值为 unlimited，这意味着它只受系统内存的限制。`-XX：MetaspaceSize` 调整标志定义元空间的初始大小如果未指定此标志，则 Metaspace 将根据运行时的应用程序需求动态地重新调整大小。

2、元空间里面存放的是类的元数据，这样加载多少类的元数据就不由 `MaxPermSize` 控制了, 而由系统的实际可用空间来控制，这样能加载的类就更多了。

3、在 JDK8，合并 HotSpot 和 JRockit 的代码时, JRockit 从来没有一个叫永久代的东西, 合并之后就没有必要额外的设置这么一个永久代的地方了。

4、永久代会为 GC 带来不必要的复杂度，并且回收效率偏低。

**方法区常用参数有哪些？**

JDK 1.8 之前永久代还没被彻底移除的时候通常通过下面这些参数来调节方法区大小。



```java
-XX:PermSize=N //方法区 (永久代) 初始大小
-XX:MaxPermSize=N //方法区 (永久代) 最大大小,超过这个值将会抛出 OutOfMemoryError 异常:java.lang.OutOfMemoryError: PermGen
```

相对而言，垃圾收集行为在这个区域是比较少出现的，但并非数据进入方法区后就“永久存在”了。

JDK 1.8 的时候，方法区（HotSpot 的永久代）被彻底移除了（JDK1.7 就已经开始了），取而代之是元空间，元空间使用的是本地内存。下面是一些常用参数：



```java
-XX:MetaspaceSize=N //设置 Metaspace 的初始（和最小大小）
-XX:MaxMetaspaceSize=N //设置 Metaspace 的最大大小
```

与永久代很大的不同就是，如果不指定大小的话，随着更多类的创建，虚拟机会耗尽所有可用的系统内存。

### 调优经验分享：如何定位CPU飙升、内存泄漏、频繁GC等问题？

##### CPU飙升问题定位

###### 现象

- 系统响应变慢，接口超时
- top命令显示Java进程CPU使用率接近100%（甚至超过100%）
- 监控系统（Zabbix、Prometheus）发出CPU告警

###### 定位步骤与工具

1. 定位高CPU进程
2. 定位高CPU线程
3. 导出线程堆栈
4. 分析堆栈
   - 死循环/空转：如`while(true)`或者错误使用for循环
   - 频繁GC：大量线程处于`VM Thread`或者`GC task thread`，说明GC本身占用了CPU
   - 锁竞争：线程阻塞在锁上，但锁竞争本身不会导致CPU高，但若锁内部执行复杂逻辑，可能导致持有锁的线程CPU高
   - 正则表达式回溯：匹配超长字符
   - 序列化/反序列化：解析大对象

###### 解决方案

- 修复代码中的死循环或低效算法
- 优化锁力度
- 调整JVM参数减少GC频率
- 使用异步处理，避免阻塞主线程

##### 内存泄漏

###### 现象

- 应用运行一段时间后，内存占用持续上升，最终OOM
- GC日志显示老年代占用不断增长，Full GC后回收效果不佳
- 监控工具展示堆内存曲线呈锯齿状但整体向上

###### 定位步骤与工具

1. 监控堆内存变化：关注E、O、`YGC`、`YGCT`、`FGC`、`FGCT`。若果O持续增长且FGC无法降低，则很有可能存在内存泄漏
2. 获取堆转储：如果JVM已经OOM，可以在启动参数中添加 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path`，让JVM在OOM时自动生成dump。
3. 分析Heap Dump
4. 常见内存泄漏模式
   - 静态集合类：如 `public static List<Object> cache = new ArrayList<>();`，不断添加对象而不清理
   - 未关闭的资源：如数据库连接、IO流、网络连接未正确关闭，导致对象被遗留
   - ThreadLocal使用不当：线程池中的线程服用，TheadLocal持有的对象未及时`remove`，导致对象被线程长期引用
   - 自定义加载类：重复加载类导致元空间泄漏
   - 内部类持有外部类引用：非静态内部类隐式持有外部类引用，导致外部类无法回收

###### 解决方案

- 确保集合类在不需要时清理
- 使用`try-with-resources`确保资源关闭
- 在线程池中使用ThreadLocal后必须显示`remove()`
- 对于元空间泄漏，检查类加载器是否被意外保留

##### 频繁GC

###### 现象

- GC日志显示Young GC或Full GC非常频繁
- 应用出现卡顿
- 监控显示GC时间占CPU比例高

###### 定位步骤与工具

1. 开启GC日志
2. 分析GC日志
   - **Young GC频率**：是否过高？间隔过短说明Eden区太小或对象分配速率过大。
   - **Young GC耗时**：是否过长？可能与对象复制成本有关。
   - **晋升情况**：每次Young GC后有多少对象进入老年代？如果晋升过快，老年代迅速填满，导致Full GC
   - **Full GC频率和耗时**：老年代或元空间是否接近满？
   - **GC后空间释放情况**：若Full GC后老年代占用仍然很高，可能存在内存泄漏或Major GC无法回收的对象。
3. 常见频繁GC原因
   - **Young GC频繁**
     - Eden区太小，对象分配速率高
     - Survivor区过小，对象过早晋升老年代
     - 大对象直接分配到老年代（-XX:PretenureSizeThreshold），导致老年代增长
   - **Full GC频繁**
     - 老年代空间不足（对象晋升过快或内存泄漏）
     - 元空间/永久代不足（-XX:MaxMetaspaceSize设置过小或类加载过多）
     - `System.gc()` 被显式调用（可通过 `-XX:+DisableExplicitGC` 禁用）
     - CMS GC出现 **并发模式失败**（Concurrent Mode Failure）或 **晋升失败**（Promotion Failed），导致退化为Serial Old GC。
   - **GC算法不当**：如大内存使用串行收集器导致长时间STW

###### 解决方案

- **调整堆大小**：-Xms、-Xmx设置合理，避免自动扩容。
- **调整代大小**：-Xmn设置年轻代大小，-XX:SurvivorRatio调整Eden和Survivor比例。
- **晋升阈值**：-XX:MaxTenuringThreshold适当提高。
- **大对象阈值**：-XX:PretenureSizeThreshold调高，避免大对象直接进老年代（但需权衡）。
- **垃圾回收器选择**：根据系统特点选择Parallel、CMS或G1。大内存（>6G）建议G1，要求低延迟可用ZGC（Java 11+）。
- **元空间设置**：-XX:MaxMetaspaceSize根据类加载需求设定。
- **禁用System.gc()**：-XX:+DisableExplicitGC（注意RMI等可能依赖）。
- **代码层面**：减少对象分配速率（如重用对象、使用基本类型、避免在循环中创建大对象）。

## **三、并发编程（30题）**

### 并发三要素：原子性、可见性、有序性。Java如何保证？

> 在多线程编程中，原子性、可见性、有序性是并发正确性的三大基石。如果这三个特性得不到保证，程序就会出现数据竞争、内存一致性错误等难以排查的问题。Java语言通过Java内存模型、关键字、锁机制和并发包提供了一系列保障

#### 原子性

##### 定义

原子性指一个或多个操作在CPU执行过程中不被中断的特性，即要么全部执行成功，要么全部不执行，不会出现执行到一半被其他线程干扰的情况

##### Java如何保证原子性

###### 通过`synchronized`关键字

`synchronized` 修饰的方法或代码块，在同一时刻只允许一个线程执行，从而保证了代码块的原子性。其原理是使用 Monitor 锁（管程），进入时加锁，退出时解锁。

###### 通过 **java.util.concurrent.locks.Lock** 接口

显式锁（如 `ReentrantLock`）提供了比 `synchronized` 更灵活的锁定操作，同样可以保证原子性。

###### 通过 **原子类（Atomic 包）**

`java.util.concurrent.atomic` 包提供了一系列原子类，如 `AtomicInteger`、`AtomicLong`、`AtomicReference` 等。它们利用 **CAS（Compare-And-Swap）** 硬件指令实现无锁的原子操作，性能优于锁。

###### 通过 **基本类型的简单赋值**（部分情况）

在 Java 中，对基本类型（除 `long` 和 `double` 外）的赋值操作是原子的。但 `long` 和 `double` 在 32 位 JVM 上可能被拆分为两次 32 位写操作，因此不是原子的。不过，现代 64 位 JVM 通常保证其原子性，但规范允许非原子，因此多线程下仍需同步或使用 `volatile` 修饰 `long/double`（`volatile` 保证原子性？其实 `volatile` 不保证复合操作的原子性，但可以保证 64 位写操作的原子性，因为 JVM 规范要求对 `volatile long/double` 的读写必须是原子的）。

#### 可见性

##### 定义

可见性指当一个线程修改了共享变量的值，其他线程能够立即看到这个修改。由于现代 CPU 有多级缓存（L1/L2/L3），每个线程可能将变量拷贝到自己的本地缓存中，导致一个线程的修改对其他线程不可见。

##### Java如何保证可见性

###### 通过volatile关键字

`volatile`是Java提供的最轻量级的同步机制。

- **保证可见性**：对`volatile`变量的写操作，会立即刷新到主内存；读操作会从主内存重新加载，而不是本地缓存
- **禁止指令重排**

###### 通过 **synchronized** 和 **Lock**

锁机制也能保证可见性。线程在加锁时，会清空工作内存中该变量的值，从而从主内存重新读取；释放锁时，会将工作内存中修改的变量刷新到主内存。JMM 规范定义了锁的获取和释放具有与 `volatile` 相同的可见性语义。

###### 通过 **final** 关键字（特殊可见性）

`final` 字段在构造器中初始化后，其他线程无需同步就能看到正确的值，前提是没有发生 `this` 引用逃逸。这由 JMM 保证：构造器结束后，`final` 字段对其他线程立即可见。

###### 通过 **原子类**（间接保证）

原子类内部通常使用 `volatile` 变量存储值（如 `AtomicInteger` 的 `value` 是 `volatile` 修饰的），因此其修改对所有线程可见。

#### 有序性

##### 定义

有序性指程序执行的顺序按照代码的先后顺序执行。但在现代编译器和处理器优化中，为了提高性能，可能会对指令进行 **重排序**（Reordering）。只要单线程语义不变，重排序是允许的，但在多线程环境下可能导致意料之外的结果。

##### Java如何保证有序性

###### 通过 **volatile** 关键字（禁止重排序）

`volatile` 变量的读写操作会插入 **内存屏障**（Memory Barrier），阻止特定类型的重排序。JMM 针对 `volatile` 制定了 **happens-before 规则**：

- 对一个 `volatile` 变量的写，happens-before 于任意后续对该 `volatile` 变量的读。
- 这保证了写操作的结果对读操作可见，并且写之前的操作不会被重排序到写之后，读之后的操作不会被重排序到读之前。

###### 通过 **synchronized** 和 **Lock**

锁的获取和释放也遵循 happens-before 规则：释放锁 happens-before 于后续获取同一个锁。这保证了临界区内的代码不会被重排序到临界区之外。

###### 通过 **happens-before 规则**

JMM 定义了 **happens-before** 规则，用于判断数据是否存在竞争。如果两个操作满足 happens-before 关系，那么第一个操作的结果对第二个操作可见，且第一个操作的执行顺序在第二个操作之前（按时间先后）。主要规则包括：

- 程序次序规则：单线程内，按代码顺序执行。
- 监视器锁规则：解锁 happens-before 加锁。
- volatile 变量规则：写 happens-before 读。
- 线程启动规则：`Thread.start()` happens-before 该线程的任何操作。
- 线程终止规则：线程的所有操作 happens-before 其他线程检测到该线程终止（如 `Thread.join()` 返回）。
- 中断规则：对线程的 `interrupt()` 调用 happens-before 被中断线程检测到中断事件。
- 对象终结规则：对象的构造完成 happens-before 其 `finalize()` 方法开始。
- 传递性：如果 A happens-before B，B happens-before C，则 A happens-before C。

###### 通过 **final** 关键字

在构造器中初始化 `final` 字段时，JMM 会禁止将 `final` 字段的写重排序到构造器之外，确保对象在构造完成后，其他线程看到的 `final` 字段是正确初始化的。

###### 通过 **原子类**（部分保证）

原子类的方法本身可能依赖 `volatile` 和 CAS，其内部操作通常不会重排序破坏语义。

### 线程池核心参数的含义？corePoolSize、maximumPoolSize、keepAliveTime、workQueue、threadFactory、handler。

#### corePoolSize（核心线程数）

##### 含义

线程池中保持存货的最小线程数，即使这些线程处于空闲状态，也不会被回收（除非设置了allowCoreThreadTimeout）

##### 行为

- 当提交一个新任务时，如果当前运行的线程数小于corePoolSize。线程池会立即创建一个新县城来处理该任务（即使有空闲线程）
- 当线程数达到corePoolSize后，新任务将被放入任务队列等待

##### 设置建议

通常根据系统的并发量和任务类型来设定

对于CPU密集型任务，核心线程数可设为CPU核心数+1；对于IO密集型任务，可适当增大，如cpu核心数*2

#### `maximumPoolSize`（最大线程数）

##### 含义

线程池允许创建的最大线程数量。它限制了线程池在极端负载下能占用的线程资源上限。

##### 行为

- 当任务队列已满且当前线程数小于 `maximumPoolSize` 时，线程池会**创建新的非核心线程**来执行任务。
- 当线程数达到 `maximumPoolSize` 且队列已满时，新任务将触发拒绝策略

##### 注意

`maximumPoolSize` 必须大于或等于 `corePoolSize`。

#### `keepAliveTime`（空闲线程存活时间）

##### **含义**

当线程池中的线程数量超过 `corePoolSize` 时，多余的空闲线程在终止前等待新任务的最长时间

##### 行为

- 如果某个线程的空闲时间超过了 `keepAliveTime`，它将被终止，直到线程数回落到 `corePoolSize`
- 该参数**仅对超过核心线程数的“非核心线程”生效**。若需对核心线程也启用超时回收，可调用 `allowCoreThreadTimeOut(true)`

##### 单位

通常与 `TimeUnit` 配合使用，如 `TimeUnit.SECONDS`。

#### `workQueue`（任务队列）

##### 含义

用于保存等待执行的任务的阻塞队列。当核心线程都在忙碌时，新提交的任务会被放入此队列。

##### 常见队列类型

- **`SynchronousQueue`**：不存储元素的队列，每个插入操作必须等待一个移除操作。适用于任务提交速率与处理速率基本匹配的场景，通常配合较大的 `maximumPoolSize` 使用。
- **`LinkedBlockingQueue`**：基于链表的无界队列（默认容量为 `Integer.MAX_VALUE`）。使用无界队列时，`maximumPoolSize` 将失效，因为队列永远不会满，可能导致内存溢出。也可指定容量变为有界队列。
- **`ArrayBlockingQueue`**：基于数组的有界队列，容量固定，有助于控制资源使用。
- **`PriorityBlockingQueue`**：支持任务优先级排序的队列

##### 选择依据

根据任务特性和系统资源选择有界或无界队列，防止任务堆积耗尽内存

#### `threadFactory`（线程工厂）

##### 含义

用于创建新线程的工厂。通过它可以自定义线程的名称、优先级、是否为守护线

##### 作用

- 默认实现为 `Executors.defaultThreadFactory()`，创建的线程属于同一个线程组，优先级为 `NORM_PRIORITY`，且非守护线程。
- 自定义 `ThreadFactory` 可以给线程起有意义的名称（方便排查问题）、设置异常处理器等。

##### 示例

```java
new ThreadFactory() {
    @Override
    public Thread newThread(Runnable r) {
        Thread t = new Thread(r, "my-pool-thread-" + counter.incrementAndGet());
        t.setUncaughtExceptionHandler((thread, throwable) -> log.error("Thread exception", throwable));
        return t;
    }
};
```

#### `handler`（拒绝策略）

##### 含义

当线程池和任务队列都满了（即线程数达到 `maximumPoolSize` 且队列已满），对新提交的任务采取的处理策略。

##### 内置实现

- **`AbortPolicy`（默认）**：直接抛出 `RejectedExecutionException`，阻止系统继续运行。
- **`CallerRunsPolicy`**：由提交任务的线程（即调用者线程）直接执行该任务。这种策略会降低新任务的提交速度，起到“慢速调节”的作用。
- **`DiscardPolicy`**：静默丢弃任务，不抛出异常也不执行。
- **`DiscardOldestPolicy`**：丢弃队列中最旧的一个未处理任务，然后重新尝试提交当前任务。

##### 自定义

可以实现 `RejectedExecutionHandler` 接口，编写个性化的拒绝逻辑（如记录日志、存入数据库等）。

#### 线程池的工作流程（参数如何协同）

结合以上参数，`ThreadPoolExecutor` 处理任务的完整流程如下：

1. 提交任务后，若当前线程数 < `corePoolSize`，创建新线程执行。
2. 若当前线程数 >= `corePoolSize`，尝试将任务放入 `workQueue`。
3. 若队列未满，任务排队等待；若队列已满且线程数 < `maximumPoolSize`，创建新线程执行。
4. 若队列已满且线程数 >= `maximumPoolSize`，执行拒绝策略 `handler`。
5. 当线程空闲时间超过 `keepAliveTime`，且线程数 > `corePoolSize`，回收多余线程。
6. 线程通过 `threadFactory` 创建，用于执行任务。

#### 实际调优建议

- **合理设置队列大小**：避免无界队列导致内存溢出，也避免队列过小导致频繁创建线程。
- **核心线程数与最大线程数**：根据任务类型（CPU/IO密集型）和系统资源预估。
- **监控与告警**：通过 `ThreadPoolExecutor` 提供的 `getPoolSize()`、`getQueue().size()` 等方法监控线程池状态，及时调整参数。
- **拒绝策略选择**：对关键业务推荐 `CallerRunsPolicy` 或自定义策略，避免任务丢失；对非关键任务可选用 `DiscardPolicy`。

### 线程池中的线程复用原理是什么？Worker如何循环从队列中取任务？

线程池能够用少量线程处理大量任务，其核心机密就在于 **线程复用**。它并不是反复创建和销毁线程，而是让线程干完一件活后，自己主动去任务队列领新活。这个机制的实现，主要围绕着 `ThreadPoolExecutor` 的一个内部类——**`Worker`**。

![](https://raw.githubusercontent.com/wangqiren/Image/main/20260311135711633.png)

#### Worker：不只是工人，更是任务的载体

`Worker` 是线程池内部对“工作线程”的抽象，它的设计很巧妙

1. **双重身份**：它同时继承了 `AQS`（抽象队列同步器）并实现了 `Runnable` 接口。作为 `Runnable`，它自己的 `run()` 方法就是线程要执行的任务；作为 `AQS`，它实现了简单的不可重入锁，用于在 `shutdown()` 等操作时标记线程的空闲状态，以便安全地中断。
2. **核心成员**：每个 `Worker` 内部有两个关键字段：`Thread thread`（执行任务的实际线程）和 `Runnable firstTask`（创建时携带的第一个任务，可能为 `null`）。
3. **创建与启动**：当我们调用 `execute()` 或 `submit()` 提交任务时，如果条件满足（比如当前线程数小于核心线程数），线程池会通过 `addWorker` 方法创建一个新的 `Worker`。在创建过程中，会用线程工厂（`ThreadFactory`）基于 `Worker` 自身（因为 `Worker` 是 `Runnable`）创建一个新线程 `t`，然后调用 `t.start()` 启动线程。线程启动后，会自动执行 `Worker` 的 `run()` 方法。

#### runWorker：决定命运的 while 循环

线程复用的精髓，全部浓缩在 `runWorker(Worker w)` 这个方法里。当 `Worker` 的线程启动后，就会进入这个方法中的一个 **无限 `while` 循环**：

1. **获取任务**：循环的第一件事，就是通过 `getTask()` 方法去获取一个可执行的任务。这个 `task` 可能是 `Worker` 创建时自带的 `firstTask`，也可能是从任务队列里取出来的。
2. **执行任务**：拿到任务后，会执行 `task.run()`。注意，这里调用的是 **`run()` 方法，而不是 `start()`**。这是一个至关重要的点，因为 `run()` 是一个同步方法调用，它不会创建新线程，而是让当前的 `Worker` 线程去执行任务代码。这正是线程复用的核心——一个线程不断执行不同任务的 `run()` 方法，而不是为每个任务都启动一个新线程。
3. **循环往复**：任务执行完毕后，会进入下一次循环，再次调用 `getTask()` 获取下一个任务。如此周而复始，实现了用一个线程处理多个任务。

#### getTask：控制生死与超时的核心

`getTask()` 方法不仅负责提供任务，还巧妙地控制了线程的生命周期，特别是非核心线程的超时回收。

它会根据当前线程池的情况，决定如何从任务队列中取任务：

- **阻塞获取**：对于核心线程（或者不允许超时的线程），会调用 `workQueue.take()`。如果队列为空，这个线程就会进入 `WAITING` 状态，一直阻塞在那里，直到有新任务加入被唤醒。这也是核心线程“保活”的秘密——它们通过 `take()` 一直等待着。
- **限时获取**：对于非核心线程（或者设置了`allowCoreThreadTimeOut`的核心线程），会调用 `workQueue.poll(keepAliveTime, TimeUnit.NANOSECONDS)`。它在尝试获取任务时，最多等待 `keepAliveTime` 这么长时间。如果在指定时间内依然没有获取到任务，`poll()` 方法会返回 `null`。
- **线程退出的信号**：当 `getTask()` 返回 `null` 时，`runWorker` 的 `while` 循环就会结束。紧接着，`processWorkerExit` 方法会被调用，将这个 `Worker` 从线程池的 `workers` 集合中移除，线程自然运行结束并被回收。

### MyISAM和InnoDB的区别？InnoDB为什么支持事务？

InnoDB的这四大特性，确实是它作为MySQL默认存储引擎的“立身之本”。它们分别从**性能**和**可靠性**两个维度，解决了数据库运行中最核心的挑战。下面我来逐一拆解它们各自解决了什么问题。

#### 插入缓冲 (Insert Buffer)：变“随机写”为“顺序写”

这个问题源于InnoDB的索引结构。当我们向一个有非聚集索引的表中插入数据时，主键索引通常是顺序增加的，写入很快。但非聚集索引的叶子节点通常不是有序的，这导致每次插入都需要随机地查找并修改磁盘上的非聚集索引页，产生大量的随机I/O，严重影响性能

##### 解决思路

InnoDB的做法是，当要插入的非聚集索引页不在内存中时，不直接去磁盘随机写入，而是先将这个更新操作记录到内存中的一个特殊结构——**插入缓冲（Insert Buffer）** 里 。插入缓冲本身也是一棵B+树，负责将多次对同一索引页的随机写入，在内存中合并为一次操作。之后，InnoDB会在后台以一定的频率，将这个缓冲中的更新操作**批量地、顺序地**合并（Merge）到实际的非聚集索引页上

##### 效果

它将大量的离散I/O请求合并，显著提升了非聚集索引的写入性能 。这个机制在后续版本中升级为**Change Buffer**，不仅能缓冲`INSERT`，还能缓冲`UPDATE`和`DELETE`操作

#### 两次写 (Doublewrite)：拯救“写一半”的数据页

这是InnoDB保证数据**可靠性**的核心。MySQL的数据页大小通常是16KB，而操作系统写磁盘是按4KB的页为单位。这就产生了一个“部分写失效（Partial Page Write）”的风险：当数据库在写入一个16KB数据页的过程中发生宕机（比如只写入了前4KB），这个数据页就损坏了，数据不完整 。更棘手的是，`redo log`（重做日志）记录的是对页的物理修改，如果页本身已经损坏，`redo log`也无法恢复 。

##### 解决思路

在写入数据文件之前，InnoDB会先把数据页拷贝到一块大小为2MB的**双写缓冲区（Doublewrite Buffer）** 中。然后，先将这2MB的数据**顺序地**写入到磁盘共享表空间中的**双写区域** 。完成这一步后，再将数据页写入到各自的数据文件中真正的目标位置 。

##### 效果

如果在写入数据文件时发生崩溃，InnoDB重启恢复时，可以从那个安全的“双写区域”中找到完好的页副本，覆盖损坏的页，然后再应用`redo log` 。这确保了即使发生“写一半”的故障，数据也不会丢失。

#### 自适应哈希索引 (AHI)：为查询打造“高速公路”的捷径

InnoDB使用B+树作为索引结构，查找数据需要从根节点遍历到叶子节点，可能需要3到4次I/O 。哈希索引的查找时间复杂度是O(1)，理论上更快，但哈希索引是无序的，不支持范围查询，所以InnoDB并没有开放给用户直接创建哈希索引 。

##### 解决思路

InnoDB会对索引的查询进行监控。如果发现某些索引页成为访问热点，并且建立哈希索引能带来性能提升，它就会**自动地、动态地**为这些热点页在内存中建立一个哈希索引 。这个哈希索引的key可以是索引键值或前缀，value指向数据页在内存中的位置

##### 效果

它为频繁的查询在B+树索引的基础上“开辟”了一条捷径，下一次查询可以直接通过哈希查找定位到数据页，避免了B+树的多次节点查找 。因为它是系统“自适应”建立的，所以称为自适应哈希索引。

#### 预读 (Read-Ahead)：让数据“提前就位”

当一个查询需要读取大量连续的数据时，如果每次都同步等待磁盘I/O，效率会非常低。

##### 解决思路

InnoDB通过两种预读算法来预测即将需要的数据

1. **线性预读**：如果观察到对表空间中某个区域的访问模式是顺序的（比如全表扫描），InnoDB会提前将该区域后续的一批数据页异步加载到缓冲池中。
2. **随机预读**：如果发现某个区域的大部分页已经被读入了缓冲池，InnoDB会推断剩下的页也很快会被用到，于是主动将它们也读入内存。

##### 效果

通过提前将数据加载到内存，当查询真正需要这些数据时，可以直接从内存中快速读取，减少了磁盘I/O的等待时间，提升了I/O效率

### 什么是聚簇索引？为什么主键最好是自增的？

#### 什么是聚簇索引？

**聚簇索引（Clustered Index）** 是一种数据存储结构，它决定了表中数据的**物理存储顺序**。在聚簇索引中，索引的叶子节点直接包含了完整的数据行，即数据行本身按照索引键的顺序存储在磁盘上。

##### InnoDB 中的聚簇索引

在 MySQL 的 InnoDB 存储引擎中，聚簇索引的实现规则如下：

- **主键作为聚簇索引**：如果你为表定义了主键，InnoDB 会自动使用主键作为聚簇索引。
- **隐式聚簇索引**：如果没有定义主键，InnoDB 会选择第一个所有列都不为空的唯一索引作为聚簇索引。
- **自动生成 rowid**：如果既没有主键也没有合适的唯一索引，InnoDB 会在内部生成一个 6 字节的隐式自增字段 `DB_ROW_ID` 作为聚簇索引。

##### 聚簇索引的特点

1. **数据即索引**：找到聚簇索引的键值，就找到了整行数据，无需再次回表。
2. **物理连续性**：逻辑上相邻的数据行在物理存储上也趋于相邻，这有利于范围查询（如 `BETWEEN`、`<`、`>` 等），因为顺序 I/O 比随机 I/O 快得多。
3. **辅助索引依赖聚簇索引**：InnoDB 中的辅助索引（二级索引）的叶子节点存储的是主键值（而不是指向行物理位置的指针）。因此，通过辅助索引查找数据时，需要先找到主键值，再通过聚簇索引回表查询完整数据，这称为**回表查询**。

##### 聚簇索引的优缺点

| 优点                                         | 缺点                                                         |
| -------------------------------------------- | ------------------------------------------------------------ |
| 范围查询性能优秀，因为数据连续存储。         | 如果主键不是有序的，插入操作可能导致页分裂，增加开销和碎片。 |
| 辅助索引可以覆盖查询时，不需要回表，效率高。 | 更新主键值（或聚簇索引列）的代价较高，因为可能需要移动行。   |
| 数据访问速度快（尤其是通过主键查找）。       | 插入速度严重依赖于主键的插入顺序。                           |

#### 为什么主键最好是自增的

##### 避免页分裂，提高插入性能

- **自增主键**：每次插入时，新行的主键值比现有最大值大，InnoDB 会直接把新行追加到聚簇索引的末尾。由于数据页是按顺序写入的，磁盘 I/O 主要是顺序写，性能极高。
- **非自增主键**（如 UUID、随机字符串）：新插入的主键值可能在现有数据的中间位置。InnoDB 需要找到目标数据页，如果该页空间不足，就会发生**页分裂**——将当前页的一半数据移到新页，并调整索引结构。页分裂涉及随机 I/O、锁竞争和索引维护，严重降低插入性能，并产生碎片。

##### 减少碎片，节省存储空间

- 自增主键的插入总是顺序的，数据页的填充率较高，空间利用率好。
- 随机主键导致的页分裂会使数据页产生大量碎片，既浪费存储空间，又增加扫描时的 I/O 次数。

##### 优化范围查询

- 自增主键本身具有顺序性，范围查询（如按时间范围或按 ID 范围）可以充分利用聚簇索引的顺序扫描，性能极佳。
- 如果主键随机，即使对主键进行范围查询，也可能需要大量随机 I/O。

##### 辅助索引更节省空间

- 由于辅助索引的叶子节点存储的是主键值，如果主键值很大（如 UUID 占用 16 字节），辅助索引的体积会显著增大，导致占用更多内存和磁盘，降低缓存效率。
- 自增主键通常为 `INT`（4 字节）或 `BIGINT`（8 字节），体积小，辅助索引也更紧凑。

##### 简化逻辑，避免分布式复杂

- 在分布式或分库分表场景中，自增主键虽然不能保证全局唯一（需结合雪花算法等），但在单机环境下实现简单，无需额外处理唯一性。
- 使用 UUID 等全局唯一主键虽然避免了分库分表的主键冲突，但会引入上述的性能问题。

##### 例外情况

虽然自增主键在大多数 OLTP 场景下是最佳选择，但在某些特殊场景下可能不适合：

- **冷热数据分离**：如日志表，可能以时间戳作为主键更自然。
- **分库分表全局主键**：需要全局唯一且有序的分布式 ID，可使用雪花算法或号段模式，生成的 ID 通常是递增的，也具备近似自增的效果。
- **业务主键明确**：例如用户表中的身份证号本身就是业务唯一标识，且查询频繁，此时直接使用业务主键作为聚簇索引可避免回表。

### 非聚簇索引和回表查询？什么是覆盖索引？

#### 非聚簇索引（二级索引）

##### 定义

非聚簇索引的叶子节点**不存储完整的行数据**，而是存储**索引列的值 + 主键值**。也就是说，通过非聚簇索引只能定位到对应主键，再通过主键去聚簇索引中查找完整行。

##### 结构特点

- 每个非聚簇索引都是一棵独立的 B+ 树。
- 叶子节点按照索引列的顺序排列，但叶子节点本身并不包含行数据，只包含索引列和主键。
- 非聚簇索引的存在不会影响表中数据的物理存储顺序（聚簇索引决定）。

##### 示例

假设有一张用户表：

```sql
CREATE TABLE user (
    id INT PRIMARY KEY,          -- 聚簇索引
    name VARCHAR(20),
    age INT,
    KEY idx_name (name)          -- 非聚簇索引
);
```

当执行查询 `SELECT * FROM user WHERE name = '张三'` 时，过程如下：

1. 在 `idx_name` 这棵 B+ 树中，找到 name='张三' 对应的叶子节点，获取主键 id。
2. 用这个 id 去聚簇索引中查找完整的行数据。

第二步就是**回表**。

#### 回表查询

##### 定义

回表是指通过非聚簇索引找到主键后，再根据主键到聚簇索引中获取完整行数据的过程。

##### 性能代价

- 如果非聚簇索引的筛选性很好（例如只返回一条记录），回表一次的成本很低。
- 但如果非聚簇索引扫描了大量记录（例如范围查询），每一条记录都需要回表，就会产生大量随机 I/O（因为聚簇索引按主键存储，回表时主键可能分散），严重影响性能。

#####  避免回表的方法

- **覆盖索引**：让索引包含查询所需的所有列，从而无需回表。
- **索引下推（ICP）**：在存储引擎层提前过滤不符合条件的记录，减少回表次数（但并没有完全消除回表）。

#### 覆盖索引

##### 定义

如果查询的所有字段（包括查询条件和返回列）都包含在同一个非聚簇索引中，那么 InnoDB 可以直接从该索引的叶子节点获取数据，而无需回表。这样的索引称为 **覆盖索引**。

##### 作用

- **消除回表**：减少一次聚簇索引查找，大幅降低 I/O 次数。
- **减少数据访问量**：索引通常比行数据小，覆盖索引可以减少存储引擎层到服务层的数据传输量。
- **优化排序和分组**：如果索引已经包含排序字段，且满足覆盖索引，可以避免 filesort。

##### 示例

仍以上面的 user 表为例，假设有一个查询

```sql
SELECT name, age FROM user WHERE name = '张三';
```

如果只有 `idx_name` 索引（只包含 name 列），那么 InnoDB 在索引中找到 name='张三' 后，叶子节点只有主键 id，无法获取 age，因此必须回表。

如果我们将索引改为：

```sql
ALTER TABLE user ADD INDEX idx_name_age (name, age);
```

现在 `idx_name_age` 的叶子节点包含 (name, age, id)。当执行上述查询时，所有需要的数据（name, age）都可以直接从索引叶子节点获得，无需回表，这就是覆盖索引。

##### 使用条件

- 查询的列必须全部包含在索引中（可以不是索引的全部列，但必须都是索引的一部分）。
- 如果是 `SELECT *`，通常无法覆盖，除非索引包含了表中所有列（这种情况很少见）。
- 覆盖索引对 `LIKE` 等模糊查询也可能生效，只要索引列覆盖了查询列即可。

#####  注意事项

- 覆盖索引并不是越大越好：如果索引列过多，会占用更多空间，降低插入/更新性能。
- 设计索引时，可以针对高频查询考虑使用覆盖索引来优化，但要在查询性能和维护成本之间权衡。

#### 总结对比

| 概念       | 定义                                             | 作用                             |
| ---------- | ------------------------------------------------ | :------------------------------- |
| 非聚簇索引 | 叶子节点存储索引列 + 主键值的索引结构            | 提高查询效率，但不直接包含行数据 |
| 回表       | 通过非聚簇索引找到主键后，再到聚簇索引中取完整行 | 必要操作，但可能成为性能瓶颈     |
| 覆盖索引   | 索引包含了查询所需的所有列，无需回表             | 消除回表，大幅提升查询性能       |

### 最左前缀原则是什么？如何利用联合索引优化查询？

#### 最左前缀原则的定义

**最左前缀原则**是联合索引在使用时需要遵循的核心规则：当查询条件中使用了联合索引的一部分列时，必须从索引的最左边开始匹配，并且不能跳过中间的列。换句话说，联合索引 `(a, b, c)` 可以被用于以下查询条件：

- `a = ?`
- `a = ? AND b = ?`
- `a = ? AND b = ? AND c = ?`
- `a = ? AND c = ?`（虽然跳过了 `b`，但 `a` 可以用到索引，`c` 无法用到索引，因为中间有断裂）

而以下查询**无法**使用该索引：

- `b = ?`（没有从最左边 `a` 开始）
- `c = ?`（没有从最左边 `a` 开始）
- `b = ? AND c = ?`（没有从最左边 `a` 开始）

#### 为什么会有最左前缀原则？

这源于 B+ 树的索引结构。联合索引的构建方式是先按第一个列排序，第一个列相同的再按第二个列排序，依此类推。因此，索引中的数据是**按索引列的顺序依次有序**的。查询时，必须按照这个顺序才能利用索引的排序特性。如果跳过了中间的列，那么后续列的排序就无法确定，因此无法继续使用索引。

#### 如何利用联合索引优化查询？

##### 根据查询条件设计索引列的顺序

- **等值条件放在最前面**：对于经常出现在 `WHERE` 子句中的等值条件（`=`），可以放在联合索引的左边，因为等值条件能过滤掉大量数据。
- **区分度高的列优先**：将区分度高的列放在前面，可以更快地缩小范围。
- **范围条件放在后面**：对于 `>`、`<`、`BETWEEN` 等范围条件，它们只能使用索引进行定位，但之后的索引列将无法继续使用。因此，应该将范围列放在索引的后面。
- **排序与分组字段**：如果查询中有 `ORDER BY` 或 `GROUP BY` 字段，尽量让它们成为索引的一部分，避免文件排序（filesort）。此时，索引的顺序应与排序顺序一致。

##### 利用索引覆盖减少回表

如果查询的列全部在联合索引中，则可以直接从索引返回结果，无需回表。这就是**覆盖索引**。例如，对于索引 `(a, b)`，查询 `SELECT a, b FROM table WHERE a = ?` 可以直接从索引中获取，无需访问聚簇索引。

##### 使用索引下推（ICP）

MySQL 5.6+ 支持索引下推，可以在存储引擎层直接过滤掉不符合条件的记录，减少回表次数。即使索引条件不完整（例如 `a = ? AND c = ?`，且索引为 `(a, b, c)`），MySQL 也会将 `c = ?` 条件下推到存储引擎，在读取索引时过滤，减少回表。

#####  避免索引失效

- **不要跳过索引中间列**：如前所述，跳过中间列会导致后续列无法使用索引。
- **避免对索引列使用函数或计算**：例如 `WHERE SUBSTR(a,1,3) = 'abc'` 会使索引失效。
- **避免隐式类型转换**：例如字符串列与数字比较，会导致类型转换，索引失效。
- **避免使用 `OR` 连接非索引列**：如果 `OR` 前后的条件不是同一个索引，可能导致全表扫描。

### 索引失效的场景有哪些？例如隐式转换、不等于、like以%开头等。

索引失效是指SQL查询无法利用索引进行快速定位，转而进行全表扫描，导致查询性能大幅下降。理解索引失效的常见场景，对于编写高效SQL和优化数据库性能至关重要。以下是基于MySQL InnoDB引擎的典型索引失效场景，附上示例和原理说明。

#### 隐式类型转换

**场景**：当查询条件中索引列的类型与传入的值类型不一致时，MySQL会自动进行类型转换，导致索引列发生函数操作，从而无法使用索引。

示例：

```sql
-- 假设表中有索引 idx_mobile (mobile)，mobile 列为 VARCHAR 类型
SELECT * FROM user WHERE mobile = 123456;  -- 数字类型，隐式转换为字符串
```

**原因**：MySQL会将 `mobile` 列的值转换为数字进行比较，相当于对索引列应用了 `CAST(mobile AS SIGNED)` 函数，索引失效。

**优化**：保持类型一致，如 `mobile = '123456'`。

#### 使用不等于（!= 或 <>）

**场景**：在查询条件中使用不等于操作符，通常会导致索引失效，因为B+树索引无法快速定位不等于的范围（需要遍历大部分数据）。

示例：

```sql
SELECT * FROM product WHERE price <> 100;
```

**原因**：不等于需要扫描全表或大部分数据，优化器通常认为全表扫描比索引更高效（尤其是当不等于值占比很小时，也可能使用索引，但一般不建议依赖）。实际上，如果选择性很高，某些情况下MySQL也可能使用索引，但稳定性较差。

**优化**：如果业务需要排除少量值，可尝试改写为 `price < 100 OR price > 100`，但一般还是建议结合业务考虑其他方案。

#### LIKE 以通配符开头

**场景**：模糊查询时，模式以 `%` 或 `_` 开头，无法利用索引的有序性进行范围匹配。

示例：

```sql
SELECT * FROM article WHERE title LIKE '%MySQL%';
```

**原因**：索引是按照字符串顺序排列的，无法快速定位以任意字符开头的字符串，只能全表扫描。

**优化**：

- 尽量避免左模糊，若业务需要可考虑全文索引（FULLTEXT）或搜索引擎。
- 如果必须用，可考虑覆盖索引（索引包含查询字段）以减少回表，但扫描依然存在。

#### 对索引列使用函数或表达式

**场景**：在索引列上应用函数、数学运算或类型转换，导致优化器无法直接使用索引，因为索引存储的是原始值，不是运算后的结果。

示例：

```sql
SELECT * FROM employee WHERE YEAR(hire_date) = 2023;   -- 函数
SELECT * FROM account WHERE balance * 1.1 > 1000;      -- 表达式
```

**原因**：索引中保存的是 `hire_date` 原始值，`YEAR(hire_date)` 需要计算后才能比较，无法利用索引。

**优化**：尽量将运算放在等式右侧，如：

```sql
WHERE hire_date >= '2023-01-01' AND hire_date < '2024-01-01';
```

#### OR 条件中包含非索引列

**场景**：`WHERE` 子句中用 `OR` 连接多个条件，如果其中一个列没有索引，则整个查询可能无法使用索引（即使其他列有索引），因为需要同时满足两个条件，而全表扫描可能是唯一选择。

**示例**：

**原因**：如果要使用 `name` 索引，只能找到 name='张三' 的记录，但还需要包含 age=25 的无索引记录，最终可能转为全表扫描。MySQL在某些情况下会使用 **索引合并（Index Merge）** 优化，但并非总是有效。

**优化**：用 `UNION` 分开查询，或为 `age` 也加上索引。

#### 联合索引未遵循最左前缀原则

**场景**：联合索引 `(a, b, c)`，如果查询条件跳过了 `a`，直接使用 `b` 或 `c`，则无法使用索引；或者跳过了中间列，如 `a = ? AND c = ?`，则只有 `a` 能用到索引，`c` 无法使用。

**示例**：

```sql
-- 索引 idx (a,b,c)
SELECT * FROM t WHERE b = 10;      -- 无法使用索引
SELECT * FROM t WHERE a = 1 AND c = 5; -- a 可用，c 不可用（中间列 b 被跳过）
```

**原因**：联合索引的B+树先按 `a` 排序，再按 `b`，最后按 `c`，跳过 `a` 或跳过中间列会导致无法利用索引的有序性。

**优化**：按查询需求调整索引列顺序，或为常用查询单独建立索引。

#### 索引列参与数学运算

**场景**：在索引列上进行加减乘除等运算，导致无法使用索引（类似于函数）。

**示例**：

```
SELECT * FROM transaction WHERE amount - 10 = 200;
```

**原因**：索引存储的是 `amount` 原始值，无法直接用于 `amount-10` 的比较。

**优化**：改写为 `amount = 210`。

#### 使用 IS NULL 或 IS NOT NULL

**场景**：对于允许为空的列，使用 `IS NULL` 或 `IS NOT NULL` 可能使索引失效，这取决于数据分布和版本。在MySQL中，如果列允许为空，`IS NULL` 通常可以使用索引（因为NULL值在索引中会被集中存放），但 `IS NOT NULL` 可能因需要扫描大量数据而被优化器放弃索引。

**示例**：

```
SELECT * FROM order WHERE remark IS NOT NULL;  -- 可能不走索引
```

**原因**：如果 `remark` 大部分都不是NULL，`IS NOT NULL` 需要扫描大部分记录，优化器可能认为全表扫描更优。

**优化**：根据实际数据分布，有时可配合其他条件来限制范围。

#### 字符集或排序规则不一致

**场景**：多表关联时，如果两个表的字符集或排序规则（collation）不同，连接条件中的字段会发生隐式转换，导致索引失效。

**示例**：

```
-- 表A 字符集 utf8，表B 字符集 utf8mb4，连接条件 A.name = B.name
```

**原因**：MySQL需要将不同字符集的值转换后才能比较，相当于对索引列应用了转换函数。

**优化**：统一字符集和排序规则。

#### 数据分布特殊，优化器选择全表扫描

**场景**：即使查询条件使用了索引列，但如果优化器估算出全表扫描比使用索引更快（例如查询结果占总行数的比例过大，或者表很小），可能会放弃索引。

**示例**：

```sql
-- 表中大部分数据 status = 1，查询 status = 1
SELECT * FROM huge_table WHERE status = 1;  -- 可能不走索引
```

**原因**：当需要返回大量数据时，使用索引需要大量随机I/O，不如顺序扫描快。

**优化**：通常这种场景无法强制使用索引，需结合业务重新设计查询或分区表

### 分页查询优化：limit 100000,10为什么慢？如何优化？

#### 为什么慢

##### 扫描大量无用数据

MySQL 执行 `LIMIT M, N` 时，实际会读取 **M + N** 行数据，然后丢弃前 M 行，只返回最后的 N 行。
例如 `LIMIT 100000, 10`，需要读取 100010 行，却只返回 10 行。前 100000 行的读取和丢弃造成了巨大的 I/O 浪费。

##### 排序成本高

如果查询包含 `ORDER BY`，且排序列没有索引，MySQL 需要将满足条件的所有数据（可能远大于 M+N 行）加载到内存或磁盘临时表进行排序，然后才能执行 `LIMIT`。深度分页时，排序数据量极大，性能灾难。

##### 回表开销大

如果索引不能覆盖查询所需的全部列，每行数据都需要回表（从聚簇索引获取完整行）。扫描 100010 行就意味着 100010 次回表，产生大量随机 I/O。

##### 锁和缓冲区压力

大量扫描会增加锁竞争和缓冲池的污染，影响其他查询。

#### 优化方案

##### 利用覆盖索引 + 延迟关联

**核心思想**：先通过索引快速定位到所需行的主键，再用主键回表获取完整数据，避免扫描大量无用行。

**示例**：

```sql
-- 原始慢查询
SELECT * FROM task ORDER BY create_time LIMIT 100000, 10;

-- 优化后
SELECT t.* 
FROM task t
INNER JOIN (
    SELECT id 
    FROM task 
    ORDER BY create_time 
    LIMIT 100000, 10
) AS tmp ON t.id = tmp.id
ORDER BY t.create_time;  -- 若需要保持排序可加，但主键关联后顺序可能被打乱，可根据业务决定
```

**说明**：

- 子查询 `SELECT id` 可以只扫描索引（假设 `(create_time, id)` 有索引），避免了回表，扫描效率高。
- 外层用 `INNER JOIN` 通过主键快速获取完整数据，回表次数仅限 10 次。

**索引要求**：需要 `(create_time, id)` 复合索引，这样子查询可以直接使用索引完成排序和分页。

##### 禁止深度分页，改为“加载更多”

**核心思想**：从产品层面限制用户翻页深度（如只允许前 100 页），或改为“无限滚动”模式（即方案二的变种）。当用户请求超过阈值时，提示“数据已到底”或使用搜索引擎。

#### 索引设计的关键

无论哪种方案，正确的索引都是基础：

- 对于 `ORDER BY col LIMIT M,N`，索引应建立在 `col` 上（如果是复合排序，则按顺序建复合索引）。
- 如果查询还有 `WHERE` 条件，尽量让索引能同时过滤和排序，避免文件排序。
- 覆盖索引能极大提升子查询效率。

### 什么是自适应哈希索引？它是如何提升性能的？

#### 什么是自适应哈希索引（AHI）

你可以把它理解为“**索引的索引**”。它的本质是一个内存中的哈希表，使用索引键的前缀作为Key，指向B+树中数据页的位置作为Value。关键在于“自适应”三个字：

- **自动构建**：它不需要人工创建，而是InnoDB自身在运行中动态完成的。
- **监控与决策**：InnoDB会持续监控对二级索引的查找。如果它发现某些索引值被**频繁地、连续地**以等值查询的方式访问（例如有资料提到，某个索引值在连续查询中被访问超过一定阈值，如17次），就会判断建立哈希索引能带来收益，从而自动为这些**热点数据**在内存中构建AHI。
- **内存驻留**：AHI完全存储在内存的`InnoDB Buffer Pool`中，因此访问速度极快。

#### AHI 如何提升性能？

它的性能提升主要来自两点：**缩短寻路路径**和**缓存热点数据**。

1. **变“遍历”为“直接定位”**：B+树索引的查找时间复杂度是`O(log n)`，需要从根节点走到叶子节点，可能涉及多次内存访问。而AHI的查找时间复杂度是`O(1)`。对于热点数据的等值查询，能直接从哈希表中拿到数据页地址，**大幅减少了CPU的计算和内存的访问次数**。
2. **缓存热点，有的放矢**：AHI并非为全表数据建立索引，它只服务于“热点”数据页。这种“好钢用在刀刃上”的策略，让它能高效利用宝贵的内存资源，只对最需要加速的查询路径进行优化。

#### 使用场景与注意事项

AHI并非在所有场景下都是“银弹”，它有自己擅长和不擅长的工作领域：

- **擅长的工作**：
  - **等值查询**：如 `WHERE name = 'ls'`，这是哈希索引的强项。
  - **热点数据**：适用于像用户中心、通行证这类有大量单行记录查询的业务。
  - **内存充足**：需要有足够大的`innodb_buffer_pool_size`来容纳AHI。
- **不适合的场合**：
  - **范围查询**：如 `WHERE age > 20`，哈希索引天生不支持。
  - **模糊查询**：如 `LIKE '%张%'`，无法使用AHI。
  - **高并发写入/Join**：维护AHI本身有开销，在某些高负载下，AHI的锁（Latch）甚至可能成为性能瓶颈。

> **如何监控与判断？**
> 你可以通过执行 `SHOW ENGINE INNODB STATUS\G`，在输出的 `SEMAPHORES` 部分查看类似 `0.00 hash searches/s, 67793.48 non-hash searches/s` 的信息。通过比较 `hash searches/s` 和 `non-hash searches/s`，可以评估AHI带来的收益。如果发现大量线程在等待AHI相关的锁，或者`non-hash searches`占比极高，可能意味着AHI帮助不大，甚至可以考虑关闭它以节省开销。

### 什么是MVCC？它的实现原理（隐藏列、Undo Log、Read View）？

#### MVCC 的核心组件

##### 隐藏列

InnoDB 的聚簇索引（主键索引）中，每一行记录都包含三个重要的隐藏列：

- **`DB_TRX_ID`**：最近一次修改（INSERT/UPDATE/DELETE）该行记录的事务 ID。事务 ID 严格递增，且每个事务有唯一 ID。
- **`DB_ROLL_PTR`**：回滚指针，指向该行记录在 Undo Log 中的上一个版本。通过该指针可以找到历史版本，形成**版本链**。
- **`DB_ROW_ID`**：隐含的自增 ID，如果表没有定义主键，InnoDB 会自动用该列作为聚簇索引。

#####  Undo Log（回滚日志）

- **作用**：记录数据修改前的旧值，用于事务回滚和 MVCC 版本读取。
- **版本链**：每次对记录进行更新（或删除），InnoDB 都会将旧版本写入 Undo Log，并通过 `DB_ROLL_PTR` 将新版本与旧版本链接起来，形成单向链表。链头是最新版本，链尾是最早版本。
- **分类**：
  - `insert undo log`：INSERT 操作产生的日志，只在事务回滚时需要，事务提交后即可清理。
  - `update undo log`：UPDATE/DELETE 操作产生的日志，用于 MVCC 构建历史版本，可能被其他事务的读操作引用，因此不能立即清理，直到没有事务需要这些旧版本时才会被 purge 线程回收。

##### Read View（一致性视图）

- **定义**：Read View 是事务进行快照读（普通 SELECT）时，生成的当前活跃事务列表的快照。它决定了哪些版本的记录对当前事务可见。
- **关键属性**：
  - `m_low_limit_id`：尚未分配的最小事务 ID，即当前已创建事务的最大 ID + 1（也有说法是“高水位”）。
  - `m_up_limit_id`：当前活跃事务的最小 ID（低水位）。
  - `m_ids`：生成 Read View 时，当前系统中所有**活跃（未提交）** 的事务 ID 列表。
- **可见性判断规则**（针对某个版本的 `DB_TRX_ID` trx_id）：
  1. 如果 `trx_id < m_up_limit_id`：表示该版本在 Read View 生成前已经提交，**可见**。
  2. 如果 `trx_id >= m_low_limit_id`：表示该版本是在 Read View 生成之后才开启的事务，**不可见**。
  3. 如果 `m_up_limit_id <= trx_id < m_low_limit_id`：需要检查 `trx_id` 是否在 `m_ids` 中：
     - 如果在 `m_ids` 中，说明该版本由尚未提交的事务产生，**不可见**。
     - 如果不在 `m_ids` 中，说明该版本在 Read View 生成前已提交，**可见**。

#### MVCC 的工作流程（以 UPDATE 为例）

假设有一个事务 T1 修改了某行记录，接着事务 T2 进行快照读。

1. **事务 T1 更新记录**：
   - 将原记录复制到 Undo Log 中，生成一个旧版本。
   - 新版本记录中 `DB_TRX_ID` 设为 T1 的 ID，`DB_ROLL_PTR` 指向 Undo Log 中的旧版本。
   - T1 尚未提交，此时最新版本（T1 修改后的版本）对其他事务不可见。
2. **事务 T2 执行 SELECT**（快照读）：
   - T2 生成一个 Read View，记录当前活跃事务列表（假设包含 T1）。
   - 顺着版本链从最新版本开始查找：
     - 检查最新版本的 `DB_TRX_ID` 是否满足可见性规则：
       - T1 的 ID 在 `m_ids` 中（未提交），因此最新版本**不可见**。
     - 通过 `DB_ROLL_PTR` 找到上一个版本（旧版本）。
     - 旧版本的 `DB_TRX_ID` 可能是更早的事务 ID，假设小于 `m_up_limit_id`，满足可见性条件。
     - T2 读取该旧版本的数据并返回。
3. **如果 T1 提交后，T2 再次 SELECT**（取决于隔离级别）：
   - 在 **READ COMMITTED** 级别下，T2 每次 SELECT 都会生成新的 Read View，此时 T1 已提交，不在 `m_ids` 中，因此 T2 能看到 T1 的修改。
   - 在 **REPEATABLE READ** 级别下，T2 只在第一次 SELECT 时生成 Read View，后续复用该视图，因此仍然看不到 T1 的修改，实现了可重复读。

#### 不同隔离级别下的 MVCC 行为

| 隔离级别        | Read View 生成时机                                 | 现象                                                        |
| --------------- | -------------------------------------------------- | ----------------------------------------------------------- |
| READ COMMITTED  | 每次 SELECT 都会生成新的 Read View                 | 每次查询都能看到已提交的最新数据（不可重复读）              |
| REPEATABLE READ | 第一次 SELECT 时生成 Read View，整个事务复用该视图 | 多次查询结果一致，避免了不可重复读和幻读（通过间隙锁+MVCC） |

注意：在 RR 级别下，MVCC 只能解决快照读的幻读问题，当前读（`SELECT ... FOR UPDATE`）仍需通过间隙锁（Next-Key Lock）来防止幻读。

#### DELETE 和 INSERT 的处理

- **DELETE**：逻辑上不是物理删除，而是标记删除。被删除的记录仍保留在版本链中，`DB_TRX_ID` 记录删除事务的 ID，同时 `DB_ROLL_PTR` 指向 Undo Log 中的旧版本。直到没有事务需要该版本时，purge 线程才会真正清理。
- **INSERT**：新插入的记录 `DB_TRX_ID` 为插入事务 ID。如果该事务未提交，其他事务通过 Read View 检查不可见；提交后可见。

#### 主从复制原理：三线程协作

整个主从复制流程可以概括为：**主库产生日志，从库拉取日志，从库回放日志**。其核心是三个线程的协同工作，具体流程如下图所示：

##### 三个核心线程的分工

- **主库 Binlog Dump 线程**：这是一个主库线程。当从库的 I/O 线程连接上来并发出请求后，它负责读取主库的二进制日志（Binlog）中的事件，并发送给从库。在读取事件时，它会短暂地对 Binlog 加锁，读取完成后立即释放，以保证并发性。
- **从库 I/O 线程**：这个线程负责连接到主库，请求 Binlog 更新。它将从主库 Binlog Dump 线程接收到的事件，写入到本地的中继日志（Relay Log）中。
- **从库 SQL 线程**：这个线程负责读取中继日志（Relay Log）中的事件，并按照顺序在从库上重新执行（回放）这些事件，从而将数据变更应用到从库，保持与主库的数据一致。

#### binlog 的三种格式：STATEMENT、ROW、MIXED

binlog 的格式决定了主库如何记录数据更改，这直接影响到从库的数据一致性、日志的存储大小和复制性能。

| 特性         | STATEMENT (语句级)                                           | ROW (行级)                                                   | MIXED (混合级)                                               |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **记录内容** | 记录修改数据的 **SQL 语句**本身                              | 记录每一行数据 **具体被修改成什么样**。例如，`UPDATE` 语句会记录所有受影响行的前后变化 | MySQL 自动判断，在 `STATEMENT` 和 `ROW` 之间动态切换         |
| **优点**     | **日志量小**，节省 I/O 和存储资源，性能较高                  | **数据一致性极高**，任何情况都能正确复制（包括使用 `UUID()`、`USER()` 等函数），是最安全的复制模式 | 结合两者优点，兼顾性能与一致性                               |
| **缺点**     | **不安全**。某些非确定性函数或上下文依赖（如 `NOW()`、`UUID()`、`LIMIT` 不带 `ORDER BY`）可能导致主从数据不一致 | **日志量大**。批量操作（如 `ALTER TABLE`、不带 `WHERE` 的 `UPDATE`）会产生巨大的 binlog，增加网络传输和磁盘 I/O 开销 | 当遇到特定情况（如包含 `UUID()`）时，自动切换为 `ROW` 模式，本质仍是 ROW 模式的日志量 |
| **适用场景** | 对性能要求极高，且能保证所有 SQL 都确定可重复执行的场景（现已较少使用）。 | **线上生产环境推荐**。对数据一致性要求严格，能容忍稍大的日志量 | 作为折中方案，可在一些对性能和一致性都有要求的场景使用。     |

关于 `MIXED` 模式的自动切换：MySQL 会在执行诸如包含 `UUID()`、`USER()`、`LOAD_FILE()` 等非确定性函数，或者更新 `NDB` 引擎表等情况下，自动将 binlog 格式从 `STATEMENT` 切换为 `ROW`，以确保复制的安全可靠。

#### 如何选择与优化

- **一致性优先，首选 `ROW` 格式**：在绝大多数生产环境中，`ROW` 格式是目前最稳妥、最被推荐的选择。它虽然可能带来稍大的日志量和网络开销，但杜绝了因 SQL 语义模糊导致的主从数据不一致风险。
- **针对性优化 `ROW` 格式的性能**：
  1. **监控并避免大事务**：`ROW` 格式下，大事务（如一次性修改上万行的 `UPDATE` 或 `ALTER TABLE`）会产生巨大的 binlog，导致主从延迟飙升。应尽量拆分大事务，或安排在业务低峰期执行。
  2. **启用 Binlog 压缩**：从 MySQL 8.0 开始，支持对 binlog 进行压缩（`binlog_transaction_compression`），可以显著减少 `ROW` 格式的磁盘和网络开销。
  3. **保证从库性能**：从库的硬件配置（CPU、I/O）不应低于主库，否则 SQL 线程回放日志的速度跟不上，会造成持续的主从延迟。

### **七、Redis与缓存（30题）**

#### Redis为什么这么快？纯内存、IO多路复用、单线程（6.0后多线程网络IO）

#### 基于内存的存储

这是 Redis 快的**基石**。Redis 是一个内存数据库，它所有的数据都存储在内存中。这意味着读写操作直接在内存上进行，没有磁盘 I/O 的寻址和传输延迟。内存的访问速度通常在纳秒级，而磁盘（即使是SSD）的访问速度在微秒甚至毫秒级，这中间有数个数量级的差距。

#### IO多路复用

虽然内存很快，但如果网络 I/O 阻塞了，性能依然会卡在网络上。Redis 使用了 **IO多路复用**技术来解决这个问题。

你可以把它想象成一个快递分拣中心：

- **阻塞 I/O**：一个分拣员只能处理一个快递，在等待这个快递入库的过程中，他什么都干不了，效率很低。
- **多线程 I/O**：一个分拣员负责一个快递，但快递很多时，需要很多分拣员，管理成本高。
- **IO多路复用**：只有一个分拣员，但他手里拿着一个**实时更新的电子看板**（`epoll`），上面显示着哪个通道有快递来了、哪个通道的快递需要发出。他只需要盯着这个看板，就可以同时处理成千上万个通道的快递，效率极高。

Redis 的 IO多路复用模块正是如此，它允许**一个线程同时监控多个客户端连接**。当某个连接的数据准备好（可读或可写）时，这个模块就会通知 Redis 线程去处理，避免了线程在等待 I/O 上浪费时间。

#### 单线程架构与 6.0+ 的多线程网络 IO

这是 Redis 最引人注目的特点，也是需要重点理解的部分。

##### 核心：单线程执行命令

Redis 处理命令（如 `GET`、`SET`）的部分是**单线程**的。这带来了几个巨大的好处：

Redis 处理命令（如 `GET`、`SET`）的部分是**单线程**的。这带来了几个巨大的好处：

- **避免了线程切换和竞争**：没有多线程的上下文切换开销，也无需考虑加锁、解锁和死锁的问题，数据一致性变得非常简单。
- **避免了锁的开销**：因为所有命令都是串行执行的，所以像 `INCR` 这种操作天然就是原子的，无需额外的同步机制。
- **代码简单可维护**：单线程模型让 Redis 的代码和底层逻辑变得非常清晰。

**单线程+多路复用**的组合，就是 Redis 的核心模型：一个线程通过多路复用技术监控大量连接，然后串行地处理就绪的事件。

##### 演进：6.0 的多线程网络 IO

既然单线程这么好，为什么 6.0 又引入了多线程？这里有一个关键点需要澄清：**Redis 6.0 的多线程只是用来处理网络 I/O 的，命令的执行依然是单线程的。**

在 6.0 之前，数据的**读取**（从socket中读入命令）和**写入**（把结果写回socket）也是由主线程完成的。当有大体积的返回结果（如 `MGET` 获取大量数据）时，网络 I/O 的读写操作会消耗不少 CPU 时间，从而阻塞其他命令的执行。

1. 主线程负责监听和分发任务。
2. I/O 线程负责与客户端进行网络数据的读写。
3. 数据读写完成后，由主线程执行命令。

这样既**保留了单线程执行命令的简单性**，又**突破了网络 I/O 的瓶颈**，进一步提升了性能。

### ZSet的底层实现是什么？跳表的特点和时间复杂度。

#### ZSet 的底层结构

| 数据规模                                           | 底层编码                                | 核心原理                                                     | 优势                                                       |
| -------------------------------------------------- | --------------------------------------- | ------------------------------------------------------------ | ---------------------------------------------------------- |
| **小数据量**(元素数 < 128 且 元素值长度 < 64 字节) | **`ziplist` (压缩列表)**                | 所有元素按`score`大小顺序紧密排列在一块连续内存中。          |                                                            |
| **大数据量**                                       | **`skiplist` (跳表) + `dict` (哈希表)** | **`zskiplist`**：负责按`score`排序，支持高效的范围操作。**`dict`**：负责通过`member`快速查找对应的`score`，时间复杂度为O(1)。 | **跳表保证有序性和范围查询性能，哈希表保证单点查询性能**。 |

#### 跳表（Skip List）的核心特点

跳表是一个“多层”的链表，可以把它想象成一个为链表专门设计的“高速公路网”，让查询可以“跳跃式”前进。

##### 1. 分层索引结构

- **底层（第1层）**：包含所有元素的有序链表。
- **上层（第2层及以上）**：是底层的“索引”，每个节点以一定的概率（Redis为1/4）出现在更高层。通过这种“空间换时间”的方式，大幅提升查询效率。

##### 2. 概率平衡（关键特性）

- 跳表不像红黑树那样通过复杂的旋转操作来强制维持平衡。它使用一个**随机函数**来决定每个节点的层高。这意味着插入和删除操作非常简单，只需修改指针，无需像平衡树那样复杂的调整。

##### 3. 独特的“跨度”设计

- Redis的跳表节点中，每一层都包含一个 `span`（跨度）字段，记录了当前节点到下一个节点之间“跳过”了多少个底层节点。这个设计使得获取元素的排名（如`ZRANK`命令）变得非常高效，无需遍历底层链表。



### Redis的过期策略有哪些？定期删除、惰性删除。内存淘汰策略有哪些（LRU、LFU、TTL等）？

#### 过期策略

当键设置了过期时间（如通过`EXPIRE`命令），Redis需要在该键过期后将其删除，以释放内存。Redis采用的是 **定期删除 + 惰性删除** 两种策略的配合，既避免了定时删除对CPU的持续消耗，又防止了过期键长期占用内存。

##### 惰性删除 (Lazy Deletion)

- **工作原理**：当客户端尝试访问一个键（如`GET`、`SET`等命令）时，Redis会先检查该键是否设置了过期时间，如果已过期则立即删除，然后返回“键不存在”的结果。
- **优点**：对CPU友好，因为删除操作只在必要时进行，不会额外消耗CPU。
- **缺点**：如果一个键过期后从未被访问，它将一直占用内存，无法被自动回收。这正是惰性删除的短板，因此需要定期删除来弥补。

##### 定期删除 (Active Deletion / Timed Deletion)

- **工作原理**：Redis每隔一段时间（默认每秒10次，即每100ms一次）随机抽取一部分设置了过期时间的键，检查它们是否过期，如果过期则删除。具体策略：
  - 从设置了过期时间的键中随机选取20个（可配置）。
  - 删除其中所有过期的键。
  - 如果过期的键占比超过25%，则重复步骤1，否则等待下一个周期。
- **优点**：通过定期扫描，能及时回收那些长期未被访问的过期键，避免内存积压。
- **缺点**：扫描频率固定，无法保证所有过期键被及时删除；同时，如果扫描期间过期键过多，可能会短暂占用CPU。
- **总结**：两种策略配合，**惰性删除保证过期键在访问时一定被清理，定期删除防止过期键永久积压**。但即使这样，如果内存写入速度过快，仍可能导致内存溢出，因此还需要内存淘汰策略。

#### 内存淘汰策略

当Redis使用的内存超过了`maxmemory`配置的上限，Redis会根据设置的内存淘汰策略，主动删除一些键来释放内存。淘汰策略的选择直接影响系统的命中率和数据一致性。

Redis提供了多种淘汰策略，从Redis 2.x到4.x逐步完善，支持LRU、LFU、随机、TTL等模式。可以通过`maxmemory-policy`配置项设置。

| 策略名称            | 作用范围           | 含义                                                         |
| ------------------- | ------------------ | ------------------------------------------------------------ |
| **noeviction**      | 所有键             | **默认策略**。内存达到上限后，不淘汰任何键，所有写请求返回错误（如`SET`、`LPUSH`等），读请求仍可正常处理。 |
| **allkeys-lru**     | 所有键             | 从所有键中，使用 **LRU（最近最少使用）** 算法淘汰最近最少使用的键。 |
| **volatile-lru**    | 设置了过期时间的键 | 从已设置过期时间的键中，淘汰最近最少使用的键。               |
| **allkeys-random**  | 所有键             | 从所有键中，随机淘汰一部分键。                               |
| **volatile-random** | 设置了过期时间的键 | 从已设置过期时间的键中，随机淘汰一部分键。                   |
| **volatile-ttl**    | 设置了过期时间的键 | 从已设置过期时间的键中，淘汰剩余存活时间最短的键（即越早过期越优先淘汰）。 |
| **allkeys-lfu**     | 所有键             | **Redis 4.0+** 新增。从所有键中，使用 **LFU（最近最不常用）** 算法淘汰访问频率最低的键。 |
| **volatile-lfu**    | 设置了过期时间的键 | 从已设置过期时间的键中，淘汰访问频率最低的键。               |

##### 各策略适用场景

- **noeviction**：适用于数据不能丢失的场景，但一旦内存写满，写入会失败，通常配合监控和扩容使用。
- **LRU系列**：适用于业务中有明显热点数据，希望保留最近访问过的键的场景。例如缓存用户会话、商品详情等。
- **LFU系列**：适用于访问频率差异较大的场景，能更精确地保留常被访问的键。例如新闻网站的热门文章，偶尔一次刷新的冷门文章会被优先淘汰。但LFU算法实现稍复杂，且需要记录访问频率。
- **随机系列**：适用于数据访问无明显热点的场景，如一些日志缓存、统计数据等，简单均匀地淘汰数据。
- **volatile-ttl**：适用于希望给数据设定存活时间的场景，优先淘汰即将过期的数据，减少过期键堆积。
